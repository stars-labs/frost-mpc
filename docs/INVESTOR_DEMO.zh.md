# 投资人演示操作手册 — 多设备 MPC 钱包

> 英文原文 / English: [INVESTOR_DEMO.md](./INVESTOR_DEMO.md)

目标：在投资人面前，由几个人各自在**自己的设备**上共同
创建一个钱包（任何一个人都不会单独持有私钥），一起完成签名，
并在物理隔离 (air-gap) 的环境下演示，同时展示多链地址——而且
**全程现场不出任何岔子**。

整体策略分为三层：

1. **彩排 + 预检 (preflight)** —— 在任何人观看之前，先用几秒钟证明整套系统可用。
2. **现场跑通多设备演示** —— 最精彩的部分。
3. **降级方案 (fallback)** —— 如果现场出现波动，就往下降一级继续进行。
   最底层那一级绝不会失败（它是完全自包含的密码学）。

---

## 0. 黄金法则（务必阅读，否则一定会吃亏）

- **每个节点都需要一个唯一的 `--device-id`。** 两个人选了相同的 id（或
  都让它默认成相同的主机名）就会在信令服务器上冲突，网状连接 (mesh) 会
  悄无声息地断掉。请提前分配好名字：`alice`、`bob`、`carol`……然后发给每个人。
- **开演前 10 分钟运行 `preflight.sh`。** 绿色 = 密码学 + WebRTC +
  网络通路都健康。红色 = 你在私下里就发现了问题，而不是在台上。
- **共用一个房间，只生成一次。** 托管服务器要求一个强 `--room`
  （≥16 个字符）。只生成一次（`uuidgen`），把完全相同的值发给每个参与者，
  并在每台设备上传入。不同房间彼此互不可见；没有房间的裸 URL 会被拒绝。
  （这同时也是租户隔离的边界——请保持每一组演示的房间私密。）
- **提前确定信令服务器**，并配置到每台设备上：
  - 现场：`--signal-server wss://panda.qzz.io --room "$ROOM"`（需要联网）。
  - 本地备份：一台笔记本运行服务器，其他人都使用 `--signal-server
    ws://<那台笔记本的局域网-ip>:9000`（需要同一 Wi-Fi，不需要外网）。请在
    会前*提前*配置好，以便能在 10 秒内切换。
- **始终准备好一台笔记本来运行 `demo-local.sh`** —— 单机降级方案。

## 1. 准备（每台设备，提前完成）

```bash
# once per device
git clone <repo> && cd mpc-wallet
nix develop                      # or have the toolchain installed
cargo build --release -p tui-node

# ONE person generates the shared room id and sends it to everyone:
ROOM=$(uuidgen)        # e.g. 7f3a9c2e-4b1d-4e8a-9c2f-001122334455

# launch the wallet (pick the id you were assigned + the SHARED room!)
cargo run --release --bin mpc-wallet-tui -p tui-node -- \
  --device-id alice \
  --signal-server wss://panda.qzz.io \
  --room "$ROOM"
```

> 托管服务器（`panda.qzz.io`）现在**要求一个强 `--room`**（≥16
> 个字符；见 #31）。同一个钱包的所有参与者都必须传入**相同**的房间。
> 裸的 `wss://panda.qzz.io`（无房间）会被拒绝。本地服务器降级方案
> （§5，独立服务器）则**不**需要房间。

2-of-3 演示中的角色：**alice、bob、carol**。（2-of-3 = 任意两人即可签名；
任何一人单独都不行。是个很好的故事。）

## 2. 预检 (preflight)（开演前 10 分钟，T-10）

在任意一台机器上：

```bash
scripts/demo/preflight.sh
# or against the server you'll actually use:
SIGNAL=wss://panda.qzz.io scripts/demo/preflight.sh
```

它会**端到端**地完整跑一遍分布式密钥生成 (DKG)（2-of-2/2-of-3/3-of-5）和
门限签名，每个只需几秒，然后检查信令服务器是否可达。全部 ✅ → 开演。
任何一个 ❌ → 先修复，或者走降级方案（§5）。这是最重要的一步——它就是
对"出事怎么办"的回答：你先在私下里确保根本不会出事。

> 为什么它值得信赖：`preflight` 使用 `mpc-wallet-cli simulate`，它会在
> 单个进程内启动一场真实的 N 节点 FROST 仪式（真实密码学、基于
> loopback 的真实 WebRTC、内嵌服务器）。这和真实客户端走的是同一条代码路径。

## 3. 现场演示 —— 4 个场景

### 场景 1 —— 在线 DKG（创建一个共享钱包）
1. **alice**：Create Wallet → 2-of-3 → 设置名称 → 设置密码 → 它会广播一个会话。
2. **bob**、**carol**：出现"session available"通知 → Join → 输入各自的密码。
3. 当 3 个人全部加入后，DKG 开始运行（几秒钟）。
4. **关键看点：** 三个屏幕都显示**同一个钱包地址**，然而**没有任何
   设备持有过私钥**——每台只持有一份密钥分片 (share)。这就是多方安全计算 (MPC)。

### 场景 2 —— 门限签名（一起签名）
1. **alice**：打开钱包 → Sign a message/tx。
2. **bob**：收到签名请求 → Approve（输入密码）。
3. 由 alice + bob 两份密钥分片生成一个有效签名。
4. **关键看点：** carol 没有参与，也不需要她（2-of-3）。可以选择性地
   演示**仅 alice 一人无法**生成签名——门限是由数学强制的，而非靠策略。

### 场景 3 —— 离线 / 物理隔离 (air-gap)（SD 卡）
1. 在 TUI 中切换到 **Offline mode**（物理隔离的 DKG/签名）。
2. 每个参与者把自己当前轮次 (round) 的数据包导出到 SD 卡 / U 盘。
3. 在机器之间物理传递这张卡；逐轮次导入。
4. **关键看点：** 密钥的生成和使用全程在机器**从未接入任何网络**的情况下
   完成——这就是冷存储 / 高安全性的故事。

### 场景 4 —— 多个钱包 + 多链地址
1. 打开 alice 的钱包详情 → 展示从同一套密钥分片派生出的
   **ETH / BTC / Solana** 地址。
2. 创建**第二个**钱包（不同门限，例如 3-of-5），以证明它不是一次性的。
3. **关键看点：** 一套 MPC 密钥 → 在每条主流链上都有地址；钱包数量不限。

## 4. 你实际在主张什么（用于问答 Q&A）

- **没有单点失陷：** 私钥从不存在于任何单一位置——不在设备上，不在
  服务器上，甚至在签名过程中也不会有哪怕一瞬间。
- **门限由密码学强制：** k-of-n 是 FROST 数学；你最多可以丢失 n−k 台设备
  仍能签名，而攻击者需要凑齐 k 份密钥分片。
- **可离线工作：** 通过可移动介质完成完整的物理隔离仪式。
- **多链：** 一套密钥分片 → ETH/BTC/Solana（以及更多）地址。

## 5. 降级方案 (fallback)（现场出现波动时）

每次只降一级。每一级都比上一级更可靠、更不依赖网络；最底层那一级是
万无一失的。

| 级别 | 触发条件 | 操作 |
|---|---|---|
| **0. 现场** | 正常 | 通过 `wss://panda.qzz.io` 进行多设备演示。 |
| **1. 本地服务器** | 外网不稳定 / panda 不可达 | 一台笔记本：`MPC_SIGNAL_BIND=0.0.0.0:9000 cargo run --release -p webrtc-signal-server`。所有人用 `--signal-server ws://<笔记本的局域网-ip>:9000` 重启各自的 TUI。同样的演示，无需外网。 |
| **2. 单台笔记本，可视化** | 某位参与者的设备出问题 | `scripts/demo/demo-local.sh` → 在一台机器上的 tmux 网格里运行本地服务器 + 3 个 TUI 节点。看起来仍然像多方协作。 |
| **3. 核选项（绝不会失败）** | 一切都着火了 | `NUCLEAR=1 scripts/demo/demo-local.sh` → `mpc-wallet-cli simulate --nodes 3 --threshold 2 --sign "…"`。约 5 秒内完成完整的 DKG + 签名 + 验证，完全自包含，打印出群组密钥 + 一个已验证的签名。"密码学就在此刻正常运行。" |

把第 2 级和第 3 级也彩排好，让切换成为肌肉记忆。

## 6. 故障排查（速查）

| 症状 | 原因 | 修复 |
|---|---|---|
| 某个节点始终不加入 / mesh 卡住 | `--device-id` 重复 | 给每个节点一个唯一 id；重启。 |
| 永远停在"Waiting for participants" | 不是所有人都连到了同一台服务器 | 确认所有人的 `--signal-server` 完全一致；检查外网 / 改用第 1 级。 |
| Join 后什么都没有 | 加入者是在广播之后才连上的 | 加入者可以重新查询会话（打开 Join 界面）；或在所有人都连上之后重新创建钱包。 |
| 显示了错误/奇怪的地址 | 构建过期 | 重新构建 release；地址由测试固定（ETH/BTC/SOL 的黄金值）。 |
| Approve 之后签名卡住 | 冷启动竞态（已修复）——确保是最新构建 | 重新构建；如果仍能复现，降到第 2/3 级。 |

---

## 7. 30 秒"出事了"决策树

1. 先前的**预检**通过了吗？如果没有——你本就不该开始；直接走第 3 级。
2. 现场跑超过约 30 秒还卡着？→ **第 1 级**（本地服务器），让所有人重连。
3. 仍然卡，或者是某台设备的问题？→ **第 2 级**（单台笔记本 tmux）。
4. 还有任何问题？→ **第 3 级**（核选项 simulate）。它一定会成功。一边重置一边
   讲解密码学原理。

现场调试绝不要超过约 30 秒。降一级，让故事继续推进，事后再修。
